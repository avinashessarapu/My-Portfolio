Two projects, picked apart in detail — architecture, decisions, and what went wrong along the way.

Public Grievance Redressal System

Full-stack web application
JavaSpring BootSpring Data JPAMySQLHTML/CSS

A full-stack system for citizens to file grievances and for admins to track and resolve them. Backed by RESTful APIs built with Spring Data JPA, role-based access control, and centralized exception handling for consistent API error responses.

↗ GitHub Live Demo — not deployed yet

Request flow: Client → Controller (REST) → Service Layer → Repository (Spring Data JPA) → MySQL. A global @ControllerAdvice handler intercepts exceptions and returns standardized error responses.

Features
  • Role-based access — separate Admin and Citizen workflows
  • REST APIs for grievance creation, tracking and resolution
  • Global exception handling for standardized errors
  • Persistent storage via Spring Data JPA + MySQL
Challenges faced
  • Designing clean role separation without duplicating controllers
  • Keeping API error responses consistent across endpoints
  • Structuring entities and relationships correctly in JPA
What I learned
  • How Spring Boot wires controllers, services and repositories together
  • Practical use of Spring Data JPA for persistence
  • Why centralized exception handling matters for real APIs

Smart Certify

Blockchain-based certificate verification system
PythonDjangoweb3.pyEthereumSmart Contracts

A decentralized application that verifies academic certificates on the Ethereum blockchain. Smart contracts handle authentication and guarantee tamper-proof storage, removing the need to trust a central certificate authority.

↗ GitHub Live Demo — not deployed yet

Request flow: Django app talks to Ethereum via web3.py → calls a smart contract to authenticate and register certificate hashes → data is stored immutably on the Ethereum ledger, making tampering detectable.

Features
  • Blockchain-backed certificate verification
  • Smart contracts for authentication logic
  • Tamper-proof, immutable certificate storage
Challenges faced
  • Connecting a Django backend to Ethereum via web3.py
  • Understanding smart contract deployment and calls for a first blockchain project
  • Reasoning about decentralization trade-offs vs a traditional DB
What I learned
  • Fundamentals of how blockchain guarantees tamper-proof records
  • How to bridge a conventional web backend with smart contracts
  • Why immutability is powerful — and where it adds real overhead

TourAlert

Real-Time Hazard Notification Backend Engine
JavaSpring BootSpring SecurityJWTSTOMP / SockJSSpring Data JPAMySQL

An event-driven public safety backend that pushes real-time transit hazard alerts to travelers the moment they're verified — no client polling required. Built around a live broadcast stream so dashboards update instantly as incidents are reported and confirmed.

↗ GitHub Live Demo — not deployed yet

Request flow: Client dashboard authenticates statelessly via Spring Security + JWT (USER/ADMIN roles) → verified incidents are pushed through a STOMP-over-SockJS broadcast channel → decoupled entity relations (user state + geographic incident metadata) are persisted via Spring Data JPA → MySQL.

Features
  • Event-driven backend for real-time transit hazard updates
  • Stateless auth with Spring Security + JWT, role-based access (USER/ADMIN)
  • Live broadcast stream via STOMP over SockJS — zero polling overhead
  • Relational schema via Spring Data JPA mapping decoupled user & incident entities
Challenges faced
  • Keeping WebSocket sessions authenticated statelessly alongside JWT
  • Designing entity relations that decouple user state from incident metadata cleanly
  • Avoiding race conditions when broadcasting to many concurrent dashboards
What I learned
  • How STOMP + SockJS deliver real-time push without polling
  • Practical stateless authentication design with Spring Security & JWT
  • Schema design for decoupled, geographically-tagged entity relationships
